# Service: Kombats.Players

## Service Purpose

Players is the source of truth for player progression and player-facing state.

It owns:
- Player
- Character
- stats relevant to player progression
- XP / level progression
- onboarding state

Players must not own:
- matchmaking queue state
- matchmaking decision logic
- battle execution state
- redis-based matchmaking projections

---

## Identity and Auth

Players uses Keycloak as the identity provider.

Rules:
- Keycloak remains the source of authentication
- Players consumes identity claims from authenticated requests
- Players must not implement its own auth server behavior
- do not duplicate identity-provider responsibilities inside Players

If the service uses first-login bootstrap / auto-provisioning patterns, preserve the existing flow and keep it idempotent.

---

## Service Boundaries

Players MUST:
- own progression updates
- own player/character state transitions
- publish integration events when player progression/profile state changes in a way other services depend on

Players MUST NOT:
- know Matchmaking storage internals
- write to Matchmaking storage
- implement matchmaking logic
- read Battle storage
- own battle result production

---

## Player Profile / Match Profile Publishing

When another service needs player state for matchmaking purposes, Players must publish integration events rather than expose internal storage.

Preferred contract style:
- snapshot-style event carrying the data Matchmaking needs

Current expected outbound event:
- `PlayerMatchProfileChangedIntegrationEvent`

Expected payload may include fields such as:
- IdentityId
- CharacterId
- Level
- IsReady
- Revision
- OccurredAt

Do not split this into many tiny cross-service events unless the system explicitly evolves that way later.

---

## Readiness Rules

For the current MVP, readiness is derived from the existing onboarding/player state.

If current domain logic defines readiness through onboarding completion, preserve that behavior rather than inventing a new concept.

Do not introduce speculative derived fields unless they are clearly required.

---

## When Players Must Publish Profile Events

Publish player profile update events after committed changes that affect matchmaking-relevant state.

Typical examples:
- character creation
- player/character setup completion
- stat allocation if it changes matchmaking-relevant projection
- XP change / level change
- onboarding transition to ready
- other committed changes that materially affect matchmaking profile data

Events must be published only after the state is successfully persisted.

---

## Battle Result Consumption

Players consumes the canonical `BattleCompleted` integration event from Battle to update progression.

Rule:
- inbound consumer must be a thin adapter
- it must route the message into existing application logic
- it must not contain XP/level business rules directly

Expected flow:
1. receive `BattleCompleted`
2. map to existing application command/use case
3. run progression update logic (XP, level, stat points, win/loss)
4. persist changes
5. publish updated `PlayerCombatProfileChanged` when needed

---

## Progression Rules

Progression logic belongs to Players.

This includes:
- XP application
- level recalculation
- revision/version updates if used by outbound contracts
- persistence of progression changes

Do not move progression logic into:
- consumers
- controllers/endpoints
- infrastructure classes
- other services

---

## Persistence Rules

- use the existing Players persistence approach
- do not introduce new persistence abstractions unless clearly justified
- keep changes localized
- do not add Redis to Players for matchmaking concerns

If outbox/inbox patterns already exist in Players, reuse them.
If not, do not introduce a large reliability subsystem unless the task explicitly requires it.

---

## Coding Conventions for Players

- follow existing CQRS conventions
- default to `internal sealed` for application-layer classes where that is the project convention
- keep query response contracts public if needed
- keep domain logic in domain/application, not transport adapters
- use file-scoped namespaces

---

## Anti-Patterns

Do NOT:
- put matchmaking logic into Players
- put battle execution logic into Players
- write directly to other services' storage
- add speculative fields such as future ranking/power-score concepts unless explicitly required
- duplicate progression logic in consumers
- redesign working application flows during simple integration wiring tasks

---

## Implementation Behavior for Tasks in Players

When working on Players tasks:

1. Analyze the existing Players flow first.
2. Reuse current commands/handlers if they already implement the required business logic.
3. Prefer thin integration wiring over business-logic duplication.
4. Keep service boundaries strict.
5. Make minimal but production-sane changes.
6. Do not add tests unless explicitly requested.
